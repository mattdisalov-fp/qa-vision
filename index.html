Function Point Studio — QA Strategy
Function Point Studio is a greenfield opportunity to build quality into the product from day one — not retrofit it later. This site outlines the QA approach, what's already been proven on the innovation team, and how it translates to Studio. The goal: catch problems in minutes, not days, so the team ships faster with confidence.

Why This Matters

Bugs found early are cheap
A bug caught in a unit test costs minutes to fix. The same bug caught in production costs days, support tickets, and customer trust. The testing pyramid puts the most checks where they're cheapest.
QA built in is invisible
When QA is architected into the system, the team barely notices it's there — tests run automatically on every PR, feedback comes in minutes, regressions are caught before merge. It's only when QA is absent that everyone notices.
AI products have unique risks
Traditional testing doesn't catch prompt quality degradation, hallucinations, or silent API cost spikes. Studio needs a testing layer designed specifically for AI behavior — something most products don't have until it's too late.
Time to market depends on confidence
The faster the team can verify a change works and hasn't broken anything else, the faster that change ships. Automated regression in CI means merging with confidence, not hope.

CI/CD Pipeline

What Runs & When
On PR: Smoke tests (~5 min). Blocks merge on failure.
On merge: Targeted regression for affected modules (~15 min).
On RC: Full regression suite (~30 min). Release gate.
Nightly: Full suite + coverage report generation.
Production: Read-only smoke only.
Approach
GitHub Actions — already in the team's workflow, no new infrastructure to manage, native Playwright support. If usage grows beyond the free tier, self-hosted runners use the exact same workflow files with zero rewrite — a 30-minute migration, not a platform change. Costs stay minimal either way.

Environment Strategy

Since we're building from scratch, we have the opportunity to engineer the QA environment alongside the product. Currently, every QA deployment requires full reconfiguration of third-party integrations — QBO, QBD, Stripe, webhooks — taking 2+ hours per cycle and repeating on every bug fix retest. Over a sprint, that's ~2 days of lost productivity per tester spent fighting infrastructure instead of testing. For Studio, persistent environments where integrations survive across deploys would eliminate this bottleneck entirely — enabling the fast turnaround and feedback loops that an end-of-year market timeline demands.

⇄ Collaboration

Focused teams, shared goal.

Function Point Studio
Leadership has been clear: optimize everyone's time on the right things. Pulling people between products means neither product gets the attention it deserves. The natural alignment is straightforward — each QA focuses where their knowledge and context runs deepest, and both products benefit from dedicated attention.
Function Point Studio
The innovation team is building Studio — QA for Studio naturally sits with the team that understands the product, the architecture decisions, and the sprint-level context. Test strategy, framework, CI/CD, execution, and release readiness all stay close to the product team.
Function Point (Core)
Core QA has years of deep product knowledge and existing coverage to build on. The 50% → 80% coverage target, regression suite, and release readiness for the core product are best served by someone who knows that codebase inside out.

Knowledge sharing happens naturally — patterns that work on one product can inform the other, and coverage gaps are filled by whoever has context and capacity. When someone's out, the other QA can step in with enough shared understanding to keep things moving.

Test Handover

The innovation framework has 148 tests across three epics. Transitioning these to the core framework is a defined process — here's how it works.

1
Deliver the handover package This sprint
Complete package with all 148 test cases — IDs, steps, expected results, priority, and automation status. Plus architecture documentation explaining each pattern (data factories, session-scoped auth, parametrized testing, retry strategies), why it exists, and what problem it solves. Both the what to test and the framework patterns needed to support it.
2
Core team implements with architectural guidance
The tests rely on patterns that don't exist in the current core framework yet — so the work is two things: expanding the framework to support these patterns (a one-time investment that benefits every future test), then translating the test logic from Python to JavaScript. Core QA leads both — they know their framework and codebase best. I help guide the architectural changes and assist with translation, especially in the early stages. How these patterns get implemented should reflect the core team's preferences.
3
Ongoing support as needed
After the handover, Core QA drives implementation on their timeline. I'm available for guidance and review — particularly on framework architecture questions. My primary focus shifts to Studio, and I can contribute additional support when capacity allows.
